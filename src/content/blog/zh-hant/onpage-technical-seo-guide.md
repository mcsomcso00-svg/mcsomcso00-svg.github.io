# On-page 與技術 SEO 完整指南（2026）

**On-page 與技術 SEO 是你完全可控的排名槓桿：清掉內容負債、鋪好內部連結、補齊 E-E-A-T 與 Schema、顧好 Core Web Vitals 與頁面元素。本指南把站內優化整理成六個模組，每個模組連到深入教學。**

## 為什麼 2026 年站內優化反而更重要

外鏈越來越難買、AI 內容讓全網供給爆炸，Google 的品質評估重心持續往「這個網站整體值不值得信」移動。站內優化的價值在於：**它 100% 在你手上**——不用等別人給連結、不用猜演算法，今天改今天生效（索引更新後）。

而站內優化在 2026 年有一個新的服務對象：AI 爬蟲。GPTBot、Google-Extended 對頁面結構的解析能力比傳統爬蟲更依賴清楚的標題階層、表格與結構化資料——你做的每一項 on-page 功課，同時在為 GEO 鋪路。

本指南按施工順序分六個模組：先清負債、再鋪連結、補信任、上 Schema、顧速度、修元素。

## 模組 1：先清內容負債——別在爛地基上蓋樓

內容負債是指那些發布超過 90 天仍拿不到點擊的頁面。它們不是中性的存在：稀釋抓取預算、拉低網站整體品質評估、讓真正該排名的頁面被平庸內容拖累。優化的第一步永遠是盤點，而不是加新內容。

實務流程：

1. 在 GSC 匯出近 90 天所有頁面的曝光與點擊。
2. 標出 0 點擊頁面，分三類處置：**合併**（主題重複）、**刪除**（無救且無連結價值）、**補料**（有曝光沒點擊）。
3. 每季重跑一次。

概念與危害見〈[內容負債是什麼](/zh-hant/blog/what-is-content-debt/)〉，GSC 的三個揪出位置與操作截圖見〈[GSC 三個地方揪出內容負債](/zh-hant/blog/gsc-content-debt/)〉，三分法的完整決策流程圖見〈[0 點擊頁面該合併、刪除還是補料](/zh-hant/blog/zero-click-pages/)〉。順帶一提：小網站通常不需要擔心 crawl budget，什麼規模才需要在意，見〈[crawl budget 是什麼](/zh-hant/blog/what-is-crawl-budget/)〉。

## 模組 2：內部連結——把權重灌進賺錢頁

內部連結是站內最被低估的槓桿。原則只有三條：

- **層級**：長尾文章（吸權重）→ 支柱頁（總整理）→ 服務頁（轉換），權重單向往上灌。
- **錨文本**：用描述性文字，不要「點這裡」；同一目標頁的錨文本要有變化。
- **微上下文**：連結前後那一兩句話要自然說明「為什麼值得點過去」，不要生硬塞連結。

一篇文章合理的內鏈數量是 3–10 條，重點是每條都有語意理由。完整策略（含孤兒頁面檢查與集群內鏈圖）見〈[內部連結策略：把權重灌進賺錢頁](/zh-hant/blog/internal-linking-strategy/)〉，集群式規劃的上游思維參考〈[語意集群怎麼建](/zh-hant/blog/semantic-cluster-guide/)〉。

## 模組 3：E-E-A-T——讓 Google 敢把你排上去

E-E-A-T（經驗、專業、權威、信任）不是單一演算法因子，而是 Google 品質評分的整體框架。站內能做的落地動作：

| 訊號 | 落地做法 |
|---|---|
| Experience | 第一手截圖、實測數據、踩坑記錄 |
| Expertise | 作者頁、署名、專業背景說明 |
| Authoritativeness | 同主題群集覆蓋、被引用與品牌搜尋 |
| Trust | 聯絡方式、關於頁、來源標註、HTTPS |

四個字母的完整拆解與 2026 年的做法，見〈[E-E-A-T 是什麼？2026 怎麼做到](/zh-hant/blog/what-is-eeat/)〉。

## 模組 4：Schema 結構化資料——用機器的語言自我介紹

Schema 是成本最低的技術 SEO 動作：一段 JSON-LD 就能明確告訴引擎「這頁是誰、何時、講什麼」。部落格網站的最低配置是三種：

```json
{ "@type": "Article", "headline": "…", "author": {…}, "datePublished": "2026-08-23" }
{ "@type": "FAQPage", "mainEntity": [ … ] }
{ "@type": "BreadcrumbList", "itemListElement": [ … ] }
```

新手最常犯的錯是標記了頁面上不存在的內容（違反規範）以及忘記驗證。從安裝到用 Rich Results Test 驗證的完整步驟，見〈[Schema 結構化資料新手實作教學](/zh-hant/blog/schema-tutorial/)〉。

## 模組 5：Core Web Vitals——速度是門票，不是加分題

Core Web Vitals 三指標的合格線：LCP ≤ 2.5 秒、INP ≤ 200 毫秒、CLS ≤ 0.1。對排名的實際影響常被誇大——它更像門票：不合格會吃虧，但從合格優化到極致的排名邊際效益很小。合理的投入順序是先讓三項全綠，然後把時間花回內容。

實測影響與優化的優先順序，見〈[Core Web Vitals 對排名的實際影響](/zh-hant/blog/core-web-vitals-ranking/)〉。圖片通常是 LCP 的最大元兇，格式選擇（WebP/AVIF）、壓縮與 alt 文字的一站式做法，見〈[圖片 SEO：alt、格式、壓縮一次搞定](/zh-hant/blog/image-seo/)〉。

## 模組 6：頁面元素——標題、meta 與 URL

最後是每篇文章都要過一遍的基本元素：

- **Title**：目標關鍵字放前半、55–60 字元內、寫給人看而不是塞字。
- **Meta description**：80–120 字、含一個具體數字或承諾，把它當廣告文案寫——它不直接影響排名，但直接影響點擊率。
- **URL**：短、英文小寫、連字號分隔、一眼看懂主題；結構反映網站層級（如 `/zh-hant/blog/slug/`）。

寫法公式與正反範例見〈[標題標籤與 meta description 怎麼寫](/zh-hant/blog/title-meta-writing/)〉；網站層級與目錄規劃的原則見〈[URL 結構與網站架構怎麼規劃](/zh-hant/blog/url-structure-seo/)〉。

## 模組 7：主題群集——單頁優化的上位思維

前六個模組解決「單頁做對」，最後一個模組解決「全站做對」。Google 評估的不只是頁面，還有網站對一個主題的**覆蓋完整度**：同一主題下你有沒有回答使用者旅程中的每個問題？這就是主題群集（topic cluster）的邏輯——一個支柱頁總覽全貌，一群深入文章各答一個子問題，彼此用內部連結織成網。

群集帶來三個站內紅利：

1. **語意訊號**：群集內文章互相引用，讓引擎確認你對主題的覆蓋深度，而不是碰巧寫了一篇。
2. **內鏈效率**：模組 2 的內部連結有了天然的結構藍圖，不用逐篇即興決定連哪裡。
3. **蠶食預防**：規劃階段就決定「一個意圖一頁」，從源頭避免多頁搶同一個關鍵字。

實務上的建置順序是：先用關鍵字研究展開主題下的所有子問題 → 分群 → 每群寫一批深入文章 → 最後寫支柱頁總整理（你現在讀的這一頁就是本站 On-page 群集的支柱頁）。規劃方法與集群架構參考〈[語意集群怎麼建](/zh-hant/blog/semantic-cluster-guide/)〉。

## 施工順序建議：90 天路線圖

把六個模組排進一季：

1. **第 1–2 週**：GSC 盤點內容負債，執行合併／刪除／補料。
2. **第 3–4 週**：鋪內部連結，消滅孤兒頁。
3. **第 5–8 週**：補 E-E-A-T 訊號＋全站上 Schema。
4. **第 9–10 週**：Core Web Vitals 修到全綠。
5. **第 11–13 週**：逐篇修 title/meta/URL，回頭驗收。

執行這張路線圖時的兩個原則：一是**每個階段都留下前後對照的數據**（清理前後的索引頁數、鋪鏈前後的孤兒頁數、修完後的 CWV 分數），下一季才知道哪些動作真的有回報；二是**不要跳關**——很多人想先做看得見的 title 與速度、把枯燥的負債盤點留到最後，結果是在髒資料上做優化，效果全部量不準。順序本身就是這張路線圖的價值。

## 自我健檢：10 分鐘快篩你的站內體質

不確定該從哪個模組下手？花 10 分鐘回答這 8 題，答「否」最多的模組就是你的優先施工區：

1. 你知道自己有幾頁發布 90 天仍 0 點擊嗎？（模組 1）
2. 全站有沒有任何一頁是沒有內鏈指向的孤兒頁？（模組 2）
3. 每篇文章有署名與作者頁嗎？（模組 3）
4. 隨機抽一頁跑 Rich Results Test，有通過的結構化資料嗎？（模組 4）
5. PageSpeed Insights 上 LCP、INP、CLS 三項全綠嗎？（模組 5）
6. 隨機抽 5 頁，title 是不是都不超過 60 字元且含目標關鍵字？（模組 6）
7. 你的每個重點主題都有支柱頁＋深入文章的群集結構嗎？（模組 7）
8. 以上檢查你多久做一次？（如果答案是「沒做過」，從模組 1 開始）

站內優化的特性是**問題不會自己消失，只會複利累積**——內容負債每季在長、孤兒頁每次發文都可能新增。所以比起一次性大掃除，更重要的是把這 8 題變成每季例行公事。

## 常見誤區：三個最浪費時間的站內迷思

最後拆掉三個讓人白忙的迷思：

1. **「關鍵字密度要達到某個百分比」**——這是十幾年前的玩法。現代引擎看的是語意覆蓋：你有沒有回答這個主題底下該回答的問題，而不是某個詞出現了幾次。刻意堆砌反而觸發過度優化訊號。
2. **「每天都要發新文章」**——頻率本身不是排名因子。10 篇深入的群集文章勝過 100 篇日更流水帳；而且對已有負債的網站，發得越多、債築得越高。先清舊帳再談新產出。
3. **「技術 SEO 做得越多越好」**——技術優化有明確的邊際遞減點：索引正常、速度全綠、Schema 就位之後，繼續鑽研技術細節的回報趨近於零。省下的時間應該回到內容與內鏈——那才是沒有天花板的部分。

判斷任何站內動作值不值得做，就問一個問題：**它有沒有讓某個真實的使用者問題被回答得更好、或被找到得更快？** 答不上來的動作，多半是在優化儀表板而不是網站。

## 常見問題（FAQ）

**Q1：On-page SEO 和技術 SEO 有什麼差別？**
On-page 指單一頁面上的內容優化（標題、內文、內鏈、meta）；技術 SEO 指網站層級的基礎建設（索引、速度、結構化資料、架構）。實務上兩者交織，所以本指南合併處理。

**Q2：站內優化多久會看到效果？**
索引層面的修正（title、Schema）通常 2–4 週反映；內容負債清理與內鏈重鋪對整體排名的影響需要 1–3 個月。

**Q3：預算有限，六個模組先做哪個？**
先做模組 1（清負債）和模組 2（內部連結）——它們零外部成本、影響全站，而且做完之後其他模組的效果才量得準。

**Q4：AI 時代還需要在意這些傳統功課嗎？**
更需要。AI 引擎引用內容前，同樣要先能抓取、解析、信任你的頁面——on-page 做得越乾淨，GEO 的起點越高。
