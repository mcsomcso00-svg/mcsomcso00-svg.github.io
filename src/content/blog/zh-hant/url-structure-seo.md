# URL 結構與網站架構怎麼規劃

**URL 結構的四個原則：短而可讀、用語意化目錄分層（如 /blog/、/services/）、slug 用英文小寫連字號、一旦發布永不更動。網站架構則遵循「任何頁面從首頁三次點擊內可達」，並讓目錄層級對應主題群集。** 架構是發布前的決策——上線後再改，每次都是一場搬家。

## 為什麼 URL 結構值得在第一天想清楚？

1. **Google 用目錄理解網站區塊**：清楚的路徑（`/zh-hant/blog/...`）讓爬蟲與 GSC 報表都能按資料夾分析。
2. **改 URL 的代價極高**：每次更動都需要 301 轉址，權重傳遞有折損風險，外部既有連結全部要靠轉址接——所以第一原則是「設計成永遠不用改」。
3. **使用者與 AI 都在讀 URL**：`example.com/blog/what-is-content-debt` 在搜尋結果、社群分享、AI 引用清單裡都比 `example.com/p?id=8471` 可信。

## URL 命名七原則

1. **小寫英文＋連字號**：`what-is-content-debt`。Google 官方建議用連字號（-）不用底線（_）；大小寫混用會製造重複網址。
2. **短，砍掉贅字**：slug 放核心語意即可，3–6 個英文單字是舒適區。`/blog/internal-linking-strategy/` 而非 `/blog/how-to-build-an-internal-linking-strategy-for-your-website/`。
3. **不放日期**：`/2026/08/17/` 讓內容看起來會過期，而且常青文更新後 URL 上的舊日期變成謊言。
4. **不放會變動的分類**：文章換分類不該換 URL——部落格文章統一放 `/blog/<slug>/`，分類用標籤或 frontmatter 管理，不進路徑。
5. **中文站 slug 建議用英文**：中文 URL 分享時會變成一長串百分比編碼（`%E5%85%A7%E5%AE%B9...`），醜且容易斷裂。中文關鍵字放 title 與內文就夠。
6. **避免參數當內容路徑**：`?id=`、`?cat=` 這類參數頁難管理、易重複；參數留給篩選與追蹤，並用 canonical 收斂。
7. **尾斜線統一**：`/blog/foo` 和 `/blog/foo/` 選一種，另一種 301 過去——否則就是全站規模的重複網址。

## 網站架構：目錄對應主題群集

以本站這種「部落格＋服務頁」的 SEO/GEO 網站為例：

```
/zh-hant/                      ← 語言根
├── blog/<slug>/               ← 所有文章（農文＋支柱頁）
├── geo/                       ← Boss 頁：GEO 服務＋工具
├── seo-audit/                 ← Boss 頁：SEO 健檢服務
└── about/                     ← 信任頁
```

要點：

- **層級淺**：內容頁最多兩層目錄。Google 官方說明 URL 深度本身不是排名因素，但每多一層，內部連結維護與爬取路徑就多一分複雜。
- **架構的「深度」由內部連結決定，不是由目錄決定**：一頁就算路徑只有一層，若從首頁要點五次才到，它就是深頁。目標：**任何重要頁面 ≤ 3 次點擊可達**。灌權重的連結設計見〈[內部連結策略](/zh-hant/blog/internal-linking-strategy/)〉。
- **主題群集靠連結成形，不必靠子目錄**：同群集文章統一放 `/blog/`、用支柱頁互鏈串起來即可；為每個群集開子目錄（`/blog/technical-seo/...`）也可以，但代價是文章換群集就得搬家。

## 扁平 vs 深層：怎麼選？

| 架構 | 適合 | 風險 |
|---|---|---|
| 扁平（所有文章在 /blog/ 下） | 部落格、內容站（≤ 數千頁） | 目錄報表無法按主題切分（可用 GSC 篩選字串補救） |
| 深層（依主題開子目錄） | 電商、大型知識庫（數萬頁+） | 分類調整成本高、容易產生重複路徑 |

多數內容網站選扁平就對了——簡單、穩定、永不搬家。

## 多語言站的路徑設計

用**語言子目錄**是多數情況的最佳解：`/zh-hant/`、`/zh-hans/`、`/en/`，配合 hreflang 標記。相比子網域（`en.example.com`）或 ccTLD（`example.tw`），子目錄讓所有語言共享同一網域權重，維護成本也最低。注意各語言的 slug 各自獨立規劃，不要用機器直譯的路徑。

## 已經上線的站想改 URL，值得嗎？

先問：現在的結構有造成實際傷害嗎（重複網址、參數地獄、報表無法分析）？

- **沒有** → 不要改。醜但無害的 URL 不值得一次搬家。
- **有** → 集中一次改到位，準備完整的一對一 301 對照表，照〈[網站搬家不掉排名的 301 轉址清單](/zh-hant/blog/301-redirect-checklist/)〉執行，別分批慢慢改。

## 常見問題（FAQ）

**Q1：URL 裡放關鍵字對排名有幫助嗎？**
有，但非常輕微——Google 官方說它是「很小的排名因素」。放的理由主要是可讀性與點擊信任感，不要為了塞關鍵字把 slug 拉長。

**Q2：.html 結尾需要拿掉嗎？**
不影響排名。新站建議用乾淨路徑（無副檔名）純粹為了美觀與一致；舊站有 .html 就留著，不值得為此轉址。

**Q3：文章 slug 發布後發現打錯字，改不改？**
剛發布（尚未收錄、無外鏈）就直接改；已收錄就權衡——錯字不影響功能的話留著，要改就必須設 301。

**Q4：URL 深度會影響 crawl budget 嗎？**
間接會：深層頁若內部連結也少，爬取優先度就低，容易卡在「Discovered - currently not indexed」。解法是連結，不是硬把目錄砍層（參見〈crawl budget 是什麼〉的需求端邏輯）。
